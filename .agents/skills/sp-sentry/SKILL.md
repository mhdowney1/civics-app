# Sentry provider guidance

Generated by `stripe projects llm-context` for Sentry, with manual cleanup.

> **Note.** The upstream `stripe projects llm-context sentry` source currently
> returns Sentry's sign-in HTML instead of markdown guidance. Use the docs links
> below until the upstream is fixed.

## Credentials you received

After provisioning `sentry/project` with the Stripe Projects CLI:

- **`SENTRY_DSN`** — Public DSN your app sends events to. Safe to expose to the
  browser. Used by `Sentry.init({ dsn })`.
- **`SENTRY_AUTH_TOKEN`** — Build-time token used to upload source maps and
  create releases. Keep secret. Only needed in CI / Vercel build env.
- **`SENTRY_ORG`** — Org slug (e.g. `civics-app`). Used at build time.
- **`SENTRY_PROJECT`** — Project slug. Used at build time.
- **`SENTRY_URL`** — Base URL of the Sentry instance (usually `https://sentry.io/`).

For Next.js, expose the DSN as `NEXT_PUBLIC_SENTRY_DSN` as well so the client
bundle can init Sentry.

## Quick start — Next.js

```bash
npm install @sentry/nextjs
```

Create four files:

- `sentry.client.config.ts` — `Sentry.init({ dsn })` for the browser
- `sentry.server.config.ts` — `Sentry.init({ dsn })` for Node runtime
- `sentry.edge.config.ts` — `Sentry.init({ dsn })` for Edge runtime
- `instrumentation.ts` — register server/edge configs, export `onRequestError`

Then wrap `next.config.ts` with `withSentryConfig(...)` so the build uploads
source maps using `SENTRY_AUTH_TOKEN`.

The interactive wizard does all of this for you:

```bash
npx @sentry/wizard@latest -i nextjs
```

In non-interactive environments (agents, CI), write the files manually — see
this repo's `sentry.*.config.ts`, `instrumentation.ts`, and `next.config.ts`
for a working example.
